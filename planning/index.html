    Planificación — Backend Engineering Program

    División Chimera — Directiva de Operaciones

    "Todo proyecto clasificado requiere una hoja de ruta. Aquí está la vuestra."

    El programa sigue una metodología Scrum adaptada: cada sprint tiene un objetivo claro, un MVP entregable y criterios de aceptación definidos. Al final de cada sprint debéis tener algo funcional que podáis ejecutar y demostrar.

    Resumen de sprints

    Sprint Nombre MVP entregable Duración estimada
    1 Setup & Primer Endpoint App Express con GET /health 2 semanas
    2 API REST en Memoria CRUD completo sin base de datos 2 semanas
    3 Persistencia con PostgreSQL CRUD con base de datos real 2 semanas
    4 Arquitectura Software Refactor a capas (Repository, DTOs) 2 semanas
    5 Autenticación JWT Login/Register + rutas protegidas 2 semanas
    6 Testing de API Suite de tests sobre endpoints 2 semanas
    7 Docker & DevOps App + BD en docker-compose 2 semanas
    8 MVP Completo Revisión final, pulido y documentación 2 semanas

    Sprint 1 — Setup & Primer Endpoint

    Duración estimada: 2 semanas

    Objetivo del sprint

    Montar el entorno de desarrollo y construir el primer endpoint funcional. Al terminar, tendréis una aplicación Express corriendo localmente que responde a peticiones HTTP.

    MVP entregable

    Una aplicación Node.js + Express con un endpoint GET /health que devuelve:

    { "status": "ok" }

    Backlog de tareas

    • [ ] Instalar Node.js y npm
    • [ ] Inicializar proyecto con npm init
    • [ ] Instalar Express como dependencia
    • [ ] Crear index.js con servidor básico
    • [ ] Implementar GET /health
    • [ ] Probar el endpoint desde el navegador y desde Postman/Thunder Client

    Conceptos que se aprenden

    • Modelo cliente-servidor
    • Qué es un servidor HTTP y cómo funciona
    • Qué es Express y por qué lo usamos
    • Estructura básica de una petición y respuesta HTTP
    • Formato JSON

    Definition of Done

    Criterios de aceptación

    • node index.js arranca sin errores
    • GET http://localhost:3000/health devuelve 200 OK con { "status": "ok" }
    • El código está en un repositorio Git con al menos un commit

    Sprint 2 — API REST en Memoria

    Duración estimada: 2 semanas

    Objetivo del sprint

    Implementar un CRUD completo sobre un recurso (ej. /missions) sin base de datos. Los datos viven en memoria (un array en el código). El objetivo es dominar el diseño REST antes de añadir persistencia.

    MVP entregable

    CRUD completo para un recurso /missions:

    Método Ruta Descripción
    GET /missions Listar todas las misiones
    GET /missions/:id Obtener una misión por ID
    POST /missions Crear una nueva misión
    PUT /missions/:id Actualizar una misión
    DELETE /missions/:id Eliminar una misión

    Backlog de tareas

    • [ ] Diseñar la estructura del recurso (campos, tipos)
    • [ ] Implementar los 5 endpoints REST
    • [ ] Manejar errores básicos (recurso no encontrado → 404)
    • [ ] Separar rutas en un fichero de router
    • [ ] Separar lógica en un controlador
    • [ ] Probar todos los endpoints con Postman/Thunder Client

    Conceptos que se aprenden

    • Métodos HTTP: GET, POST, PUT, DELETE
    • Códigos de estado HTTP (200, 201, 404, 400, 500)
    • Diseño REST: recursos, rutas, verbos
    • Parámetros de ruta (:id) y cuerpo de la petición (body)
    • Separación en routers y controladores

    Buena práctica

    Organizad el código desde el principio: routes/, controllers/. Refactorizar después es costoso.

    Definition of Done

    Criterios de aceptación

    • Los 5 endpoints responden con los códigos de estado correctos
    • Un recurso inexistente devuelve 404
    • El código está separado en rutas y controladores
    • Todos los endpoints probados y documentados en Postman

    Sprint 3 — Persistencia con PostgreSQL

    Duración estimada: 2 semanas

    Objetivo del sprint

    Reemplazar el array en memoria por una base de datos PostgreSQL real. Los datos deben sobrevivir a reinicios del servidor.

    MVP entregable

    El mismo CRUD del Sprint 2 pero con PostgreSQL como capa de persistencia. Al reiniciar el servidor, los datos siguen ahí.

    Backlog de tareas

    • [ ] Instalar PostgreSQL localmente
    • [ ] Crear base de datos y tabla para el recurso
    • [ ] Instalar el driver pg (node-postgres)
    • [ ] Configurar la conexión a la BD
    • [ ] Reescribir los controladores para usar queries SQL
    • [ ] Manejar errores de BD (conexión fallida, constraint violation)
    • [ ] Usar variables de entorno para la cadena de conexión

    Conceptos que se aprenden

    • SQL básico: SELECT, INSERT, UPDATE, DELETE
    • Tablas, columnas, tipos de datos, claves primarias
    • Driver pg y conexión a PostgreSQL desde Node.js
    • Variables de entorno con dotenv
    • Gestión de errores asíncronos

    Atención

    Nunca hardcodéis credenciales de base de datos en el código. Usad siempre variables de entorno.

    Definition of Done

    Criterios de aceptación

    • Los datos persisten tras reiniciar el servidor
    • La cadena de conexión se lee de una variable de entorno
    • El fichero .env está en .gitignore
    • Las queries SQL están correctamente parametrizadas (sin SQL injection)

    Sprint 4 — Arquitectura Software

    Duración estimada: 2 semanas

    Objetivo del sprint

    Refactorizar la aplicación para adoptar una arquitectura por capas profesional: separar la lógica de acceso a datos (Repository), las reglas de negocio (Service) y los objetos de transferencia de datos (DTOs).

    MVP entregable

    La misma funcionalidad del Sprint 3, pero con el código organizado en capas bien definidas.

    Backlog de tareas

    • [ ] Crear capa repositories/ con acceso directo a BD
    • [ ] Crear capa services/ con lógica de negocio
    • [ ] Definir DTOs para entrada y salida de datos
    • [ ] Refactorizar controladores para que solo orquesten
    • [ ] Asegurarse de que cada capa tiene una sola responsabilidad

    Conceptos que se aprenden

    • Arquitectura MVC y arquitectura por capas
    • Repository Pattern: abstraer el acceso a datos
    • DTO (Data Transfer Object): separar la representación interna de la API
    • Principio de responsabilidad única (SRP)
    • Por qué la arquitectura importa cuando el proyecto crece

    Regla de oro

    Un controlador no debería contener lógica de negocio ni queries SQL directamente.

    Definition of Done

    Criterios de aceptación

    • Existe una capa Repository que encapsula todas las queries
    • Los controladores solo llaman a servicios, no a la BD directamente
    • Los DTOs validan los datos de entrada antes de procesarlos
    • La funcionalidad del Sprint 3 sigue funcionando exactamente igual

    Sprint 5 — Autenticación JWT

    Duración estimada: 2 semanas

    Objetivo del sprint

    Añadir autenticación al sistema. Los usuarios podrán registrarse, hacer login y obtener un token JWT que les permita acceder a rutas protegidas.

    MVP entregable

    • POST /auth/register — Crear cuenta de usuario
    • POST /auth/login — Obtener token JWT
    • Rutas de misiones protegidas: requieren token válido en la cabecera

    Backlog de tareas

    • [ ] Crear tabla users en PostgreSQL
    • [ ] Implementar POST /auth/register con hash de contraseña (bcrypt)
    • [ ] Implementar POST /auth/login que devuelve un JWT
    • [ ] Crear middleware de autenticación que valide el token
    • [ ] Proteger las rutas del recurso principal
    • [ ] Implementar roles básicos (admin vs. usuario)

    Conceptos que se aprenden

    • Autenticación vs. autorización
    • Hashing de contraseñas con bcrypt
    • JWT: estructura, firma, expiración
    • Middleware en Express: cómo y cuándo usarlo
    • Roles y permisos básicos

    Seguridad crítica

    Nunca almacenéis contraseñas en texto plano. Siempre usad bcrypt u otro algoritmo de hashing seguro.

    Definition of Done

    Criterios de aceptación

    • Un usuario no autenticado recibe 401 Unauthorized al acceder a rutas protegidas
    • El token JWT expira correctamente
    • Las contraseñas se almacenan hasheadas en la BD
    • El secreto JWT se lee de una variable de entorno

    Sprint 6 — Testing de API

    Duración estimada: 2 semanas

    Objetivo del sprint

    Escribir una suite de tests automatizados que cubra los endpoints principales. Los tests deben poder ejecutarse con un solo comando.

    MVP entregable

    Suite de tests con Jest + Supertest que cubre los endpoints CRUD y autenticación.

    Backlog de tareas

    • [ ] Instalar Jest y Supertest como dependencias de desarrollo
    • [ ] Configurar Jest en package.json
    • [ ] Escribir tests unitarios para la lógica de los servicios
    • [ ] Escribir tests de integración para los endpoints REST
    • [ ] Configurar una BD de test separada
    • [ ] Añadir script npm test al proyecto

    Conceptos que se aprenden

    • Tests unitarios vs. tests de integración
    • Jest: describe, it/test, expect, mocks
    • Supertest: simular peticiones HTTP en tests
    • Aislamiento de tests: setup y teardown
    • Por qué el testing importa en proyectos profesionales

    Cobertura mínima

    Cubrid al menos los happy paths y los casos de error más comunes (recurso no encontrado, credenciales incorrectas).

    Definition of Done

    Criterios de aceptación

    • npm test ejecuta todos los tests sin errores
    • Los tests cubren los endpoints CRUD y los endpoints de autenticación
    • Los tests son independientes entre sí (no dependen del orden de ejecución)
    • La BD de test se limpia entre ejecuciones

    Sprint 7 — Docker & DevOps

    Duración estimada: 2 semanas

    Objetivo del sprint

    Contenerizar la aplicación y la base de datos con Docker. Al finalizar, cualquier miembro del equipo podrá levantar el entorno completo con un solo comando.

    MVP entregable

    Un docker-compose.yml que levante la aplicación Node.js y PostgreSQL juntas con docker-compose up.

    Backlog de tareas

    • [ ] Crear Dockerfile para la aplicación Node.js
    • [ ] Crear docker-compose.yml con servicios app y db
    • [ ] Configurar variables de entorno en docker-compose
    • [ ] Añadir health check para esperar a que PostgreSQL esté listo
    • [ ] Documentar el proceso de arranque en el README
    • [ ] Verificar que los tests pasan dentro del contenedor

    Conceptos que se aprenden

    • Contenedores vs. máquinas virtuales
    • Dockerfile: instrucciones para construir una imagen
    • Docker Compose: orquestar múltiples contenedores
    • Variables de entorno en contenedores
    • Redes Docker: cómo se comunican los contenedores entre sí

    Ventaja clave

    Con Docker, el "en mi máquina funciona" desaparece. El entorno es reproducible para todo el equipo.

    Definition of Done

    Criterios de aceptación

    • docker-compose up levanta la app y la BD sin errores
    • La aplicación es accesible en localhost:3000 tras el arranque
    • Las variables de entorno se configuran desde fuera del contenedor
    • El fichero .env no está en la imagen Docker

    Sprint 8 — MVP Completo

    Duración estimada: 2 semanas

    Objetivo del sprint

    Sprint de cierre: revisión final, pulido del código, documentación completa y preparación para despliegue. Al terminar, tendréis un backend profesional listo para producción.

    MVP entregable

    Backend completo, documentado y desplegable:

    • Código limpio y revisado
    • README con instrucciones de instalación y uso
    • Variables de entorno documentadas
    • Pipeline CI/CD básico (opcional: GitHub Actions)
    • Posible despliegue en Railway, Render o similar

    Backlog de tareas

    • [ ] Revisión de código: eliminar código muerto, mejorar nombres
    • [ ] Asegurarse de que todos los errores están manejados correctamente
    • [ ] Escribir o actualizar el README
    • [ ] Documentar todos los endpoints (puede ser con Swagger/OpenAPI)
    • [ ] Configurar un pipeline CI que ejecute los tests automáticamente
    • [ ] (Opcional) Desplegar en un servicio cloud gratuito

    Conceptos que se aprenden

    • Buenas prácticas de código y mantenibilidad
    • Documentación de APIs (README, Swagger)
    • CI/CD: qué es y por qué automatizar
    • Deploy: diferencias entre entornos (desarrollo, staging, producción)
    • Code review: cómo dar y recibir feedback técnico

    Deuda técnica

    Este sprint es el momento de pagar la deuda técnica acumulada. No lo useis para añadir funcionalidades nuevas.

    Definition of Done

    Criterios de aceptación

    • El proyecto arranca desde cero siguiendo únicamente el README
    • Todos los tests pasan en CI
    • El código ha pasado por al menos una revisión entre compañeros
    • La API está documentada (mínimo los endpoints principales)
